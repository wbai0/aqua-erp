# 慈德ERP 库存系统 · 核心逻辑说明

> 本文是对现有桌面版 ERP(SQL Server 后端)库存模块运行逻辑的梳理,
> 于 2026-07 的一次数据事故排查中,通过**核对生产库真实数据 + 阅读数据库里的存储过程/函数定义**逆向总结而成。
> 标注「✅ 已核实」的是有数据/代码依据的结论;「🔶 推断」的是根据现象合理推断、但未 100% 证实的部分。
>
> 环境:生产库为 **SQL Server 2008 R2**,数据库 `cide_main`。

---

## 0. 一句话总览

**这套软件的库存和批次结余,几乎全部是从「单据」实时计算出来的,而不是从库存快照表读的;而且大部分计算不依赖「审核」状态。** 理解这一条,就理解了 90%。

---

## 1. 最重要的三条原则(✅ 已核实)

### 1.1 库存从单据现算,不看审核状态
- 「即时库存」= 该物料/仓库下 **所有入库明细数量之和 − 所有出库明细数量之和**,**未审核的单据也计入**。
- 证据:生产库里几乎所有单据都是「未审核」,但桌面端「即时库存」照样显示正确的在库数(如某半成品 61200 千克);而按单据 `Σ入库 − Σ出库` 算出来的数,与桌面端显示一致。

### 1.2 快照表基本不用,常年为 0
- `t_stock_material`(物料级库存快照)和 `t_material_batch` 的 `in/out/left_quantity`(批次数量),在正常工作状态下**全是 0**。
- 说明桌面端**不依赖这两张表的数量字段做展示**,而是每次从单据现算。这两张表更像是历史遗留/理论上由「审核」维护、但实际未被使用的字段。

### 1.3 「审核」在这套部署里基本没被使用
- 桌面端的 **生产入库审核本身就是坏的**(点审核抛 .NET 空引用 `未将对象引用设置到对象的实例`,记录在 `t_s_log_exception`)。
- 因此正常运作状态就是「所有单据长期保持未审核」。审核与否不影响库存、批次、下拉的正常使用。
- 🔶 推断:桌面端的 `audit_time` 字段几乎不被写入(历史单据该列全空),可作为「是否被非原生程序动过」的指纹。

> ⚠️ 事故教训:任何**外部程序**若按「常规 ERP」的思路去「审核」单据、并往 `t_stock_material` / `t_material_batch` 写数量、或造出自己格式的批次行,都会与这套「靠单据现算」的模型**不兼容**,导致桌面端行为异常。本套系统的正确用法是**只读**,或严格复刻桌面端逻辑。

## 3. 物料编码规则(✅ 已核实)

物料编码形如 **`类别.供应商.序号`**,例如 `BCP.WXF.001`:

- **第一段 = 类别**(对应 `t_a_material_type`):
  - `WYL` 外购原材料(即「毛料」)、`BCP` 半成品、`CP` 成品库、`XP` 虾片、`BFC` 包辅材、`BZW` 包装物
- **第二段 = 供应商**:`WXF`=王显峰、`MY`=牧怡、`MRL`=马如令、`TK`/`XH` 等
- **第三段 = 序号**

所以「供应商 / 类别」这类信息其实**内嵌在物料编码里**,同一物料的供应商是固定的。

## 4. 单据类型(字典,✅ 已核实)

**入库 `t_a_in_type`**(取号前缀见 `server/src/cloud/writes.ts`):
- `01` 采购入库(CGRK) · `13` 生产入库(SCRK) · `01_1` 虾片采购入库 · `05` 包装物入库 · `06` 期初入库 · `07` 暂存库签收 · 生产/报废再生料入库、销售退货入库、采购退货等

**出库 `t_a_out_type`**:
- `01` 生产出库(SCCK) · `02` 销售出库(XSCK) · `13` 毛料出库(MCK) · `03` 盘亏 · `04` 报损 · `10` 开发领料 · `12` 生产退库 · 销售退库、虾片销售出库等

## 5. 库存计算(即时库存,✅ 已核实)

```
某物料在某仓库的即时库存
  = Σ(该物料该仓库的入库明细.quantity)
  − Σ(该物料该仓库的出库明细.quantity)
（所有单据,不分审核状态）
```

- 「即时库存-按批次」则再按 **检索码 `s_code`** 拆分:每个 `s_code` 的结余 = 该检索码下 `Σ入库 − Σ出库`。
- 展示字段(供应商/产地/水分/包装规格等)来自 `t_a_material` 档案 + `t_material_batch`(取该物料一条代表批次的供应商/产地)。
- 「包装数量」= 库存数量 ÷ 包装规格(`pack_spec`)。

---

## 6. 批次逻辑(核心,✅ 已核实 —— 本次事故的关键)

### 6.1 批次结余靠「明细上的 `material_batch_id`」现算
数据库里两个标量函数决定一切:

- `f_get_material_batch_id_in_quantity(@material_batch_id)`
  = `Σ t_stock_in_detail.quantity WHERE material_batch_id = @批次`  ——「该批次入库量」
- `f_get_material_batch_id_in_used_quantity(@material_batch_id)`
  = `Σ t_stock_out_detail.quantity WHERE material_batch_id = @批次`  ——「该批次已领用量」

**批次结余 = 入库量 − 已领用量**,两者都是**实时从明细的 `material_batch_id` 求和**得来,**不读 `t_material_batch.left_quantity`**。

### 6.2 「选择出库批次」下拉的填充逻辑
- 生产出库单里的「选择出库批次」下拉,列出的是 `t_material_batch` 中 **结余(按上面函数算)> 0** 的批次。
- 因此:**只要入库明细的 `material_batch_id` 正确指向批次,下拉就有数据;一旦这个链接丢失(被置空),函数算出来入库量=0、结余=0,下拉就空了。**
- 🔶 本次事故正是外部程序把入库明细的 `material_batch_id` 清空,导致下拉全空、货发不出;按 `物料+检索码` 把链接重新接回批次后恢复。

### 6.3 批次身份
- `t_material_batch` 一行 = 一个批次,主键 `material_batch_id` 形如 `20260206_010252_BCP.MRL.003`(时间戳_物料)。
- 按 `(material_id, s_code)` 唯一(✅ 实测无重复),所以「物料 + 检索码」可唯一定位批次。

## 7. 两种出库方式(✅ 已核实)

| | 生产出库 / 半成品·成品出库 | 毛料出库(领料) |
|---|---|---|
| 挑选依据 | **按批次**:选 `t_material_batch` 里有结余的批次 | **按入库单**:从「选择入库单」里挑一条毛料入库明细领用 |
| 链接字段 | 明细 `material_batch_id` → 批次 | 明细 `in_id` / `in_detail` → 对应入库明细 |
| 结余口径 | 批次入库量 − 批次已领用量(见 6.1) | 该入库明细数量 − 已被出库领用量 |
| 现象 | 「请选择出库批次」下拉 | 「选择入库单」下拉 |

## 8. 生产流转与溯源(✅ 已核实)

- 物料流:**毛料(WYL)→ 加工 → 半成品(BCP)→ 成品(CP)**。毛料被领用消耗,产出半成品;半成品再出库/加工成成品。
- **检索码 `s_code`(如 `WXF.XH.2501`)贯穿全链路**:毛料入库、毛料出库(投产)、生产入库(半成品)都带同一个 `s_code`,由此可做批次溯源(毛料哪一车 → 变成哪批半成品 → 成品)。

---

## 9. 关键存储过程 / 函数速查(✅ 已核实存在)

- `f_get_material_batch_id_in_quantity` / `f_get_material_batch_id_in_used_quantity` —— 批次入库量 / 已领用量(见 6.1)
- `f_get_s_code_by_material_batch_id` —— 由批次反查检索码
- `f_get_material_quantity` / `f_get_material_in_quantity` / `f_get_material_out_quantity` —— 物料级库存/进/出量
- `f_get_detail_out_quantity` —— 明细已出库量
- `p_e_get_counter_batch` —— (名字带 batch,实为**取号器**,与批次结余无关,勿混淆)

> 一整套 `f_get_material_*` 函数说明:桌面端的数字几乎都是**函数现算**,不是存字段。

## 10. 对接建议(给 Web 只读查看器 / 任何外部程序)

1. **只读**。不要对生产库做审核/写入。本套系统靠单据现算,外部写快照表/批次会破坏它。
2. 算库存:`Σ入库明细 − Σ出库明细`(按物料或按 `s_code`),**不要按审核状态过滤**(否则会是空的,因为几乎都未审核)。
3. 算批次结余:用 `material_batch_id` 关联明细求和(复刻 6.1 的两个函数),**不要读 `t_material_batch.left_quantity`**(常年为 0)。
4. 展示字段:类别/包装规格等取 `t_a_material`;供应商/产地取 `t_material_batch` 代表批次或从物料编码解析。
5. 若必须动数据:先在**本地副本**复刻+验证,再在生产上**单事务**执行,并**先做完整备份**。

## 附:本次事故要点(备忘)

- 外部程序把大量单据「批量审核」并写入生产库,造出 `MB` 前缀的批次、往 `t_stock_material`/`t_material_batch` 写数量、把单据改成已审核 —— 与桌面端「靠单据现算」的模型冲突。
- 清理时又把入库明细的 `material_batch_id` 一并置空,导致 6.2 的下拉全空。
- 修复路径:① 按 `audit_time` 指纹回滚所有被误审的单据、删除 `MB` 批次;② 按 `(material_id, s_code)` 把 `material_batch_id` 重新接回原生批次,恢复「选择出库批次」。
- 根因结论:**这套系统必须当作「只读、靠单据现算」来对待。**
